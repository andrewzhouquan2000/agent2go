## 15:57 - 用户深度点评（致命问题指出）

### 问题 1: 收费模型缺失
**用户批评**: "文档中完全没有提及如何赚钱。对于小企业主（老张）来说，他们对价格极其敏感。你的成本是高昂的（GPT-4o/Claude3 API 调用）。如果你按 Token 收费，用户会觉得不可控；如果你按 Seat 收费，用户会觉得'我没干活也要付费'。"

**用户建议**: "必须设计基于价值的收费模式。例如：按生成的 Listing 数量收费，按成功发布的笔记收费，而不是按 Token 或时间收费。或者按月*agent 费用"

### 问题 2: Workflow Builder 劝退设计
**用户批评**: "你的目标用户是'小企业主'和'内容创作者'，不是程序员。让他们去画布上拖拽节点（Node）、定义Edges、配置 JSON，是反人类的。绝大多数小白用户连 Zapier 都用不明白。"

**用户建议**: "把'Workflow Builder'变成'高级模式'，默认是'向导模式'（Wizard Mode）。"

### 问题 3: Templates 优先级本末倒置
**用户批评**: "你提供的'预设模板'是唯一的救命稻草，但文档中模板功能被归类为 P2 增强功能，这是本末倒置。"

**用户建议**: "把 Templates 从 P2 提到 P0。用户不是来'造车'的，他们是来'开车'的。"

### 问题 4: Task 层级过度设计
**用户批评**: "你设计了 Parent Task + Child Tasks。这在技术上是完美的（符合行业最佳实践），但对于'简单任务'（如写一篇小红书）来说，这是过度工程化。"

**用户建议**: "对小白用户隐藏'Parent/Child'概念，只显示'任务'。对高级用户（开发者）才开放底层的编排视图。"

### 问题 5: Prompt Editor 鸡肋化
**用户批评**: "真正的小企业主根本不知道什么是 System Prompt，什么是变量。这个功能应该完全由'模板'覆盖，或者通过自然语言配置（NLP to Prompt）来实现，而不是让他们手写代码式的 Prompt。"

### 问题 6: Vercel Serverless Timeout 风险
**用户批评**: "AI 任务（特别是复杂的 Workflow）可能会超过 Vercel Serverless Function 的执行时间限制（通常 10-60 秒）。如果一个亚马逊 Listing 生成任务跑了 2 分钟，Vercel 会强制 Kill 掉进程。"

**用户建议**: "必须引入任务队列（Queue）机制（如 Upstash Redis Queue 或 BullMQ），将长任务异步化。"

### 问题 7: Usage 表缺失
**用户批评**: "你需要一张独立的 user_usage 表来实时记录用户的 Token 消耗或生成次数，这是计费的核心依据，目前文档中缺失。"

**用户建议**: "加入 Queue 系统 + Rate Limit + Usage 表"
